# DevContainer & credential setup (post-registration)

**What it does.** Once a runtime container registers over the downstream WebSocket, the
controller — not the orchestrator — drives one-time environment and credential provisioning
into that runtime, using data captured when the runtime was spawned. This is how a fresh
sandbox gets its coding-agent CLI installed, its credentials seeded, and its repos cloned.

**How a peer (the runtime) interacts.** Register first; then receive a fixed-order burst of
DOWN commands the runtime must handle. Only the first is always sent; the rest are
conditional on what the controller holds for this runtime:

1. `setup_devcontainer` — always sent (when setup is pending). Carries a generated
   devcontainer config plus an OPTIONAL `auth_token`.
2. `setup_codex_credentials` — only when a Codex ChatGPT-subscription `auth.json` bundle
   was provisioned. Carries the opaque bundle to write.
3. `setup_coding_agent_credential` — only when the orchestrator resolved a coding-agent
   credential. The single generic credential carrier (see below).
4. `setup_git_clone` — only when the spawn included git config. Carries the repo list.

**Observable behavior.** All four are pushed WITHOUT the controller awaiting a correlated
reply (fire-and-forget). The runtime performs each action and, for devcontainer setup,
reports back with a `setup_complete` message (`{success, message, version?}`); on
`success:true` the controller marks the runtime ready and forwards the event upstream. The
whole burst fires only once — gated on a "setup pending" flag set at spawn; a runtime that
reconnects with setup already done receives none of it.

**Contract.**
- `setup_devcontainer` in: `{devcontainer, auth_token?}`. The devcontainer config is
  generated by the controller by detecting the OS from the runtime's base image
  (alpine / ubuntu / debian / centos / fedora, else debian) and emitting an OS-specific
  `postCreateCommand` that installs the coding-agent CLI via the matching package manager
  (apt / apk / dnf).
- `setup_codex_credentials` in: `{auth_json}` — opaque bundle, written verbatim.
- `setup_coding_agent_credential` in: `{auth_type, fields, config?, connection_uuid?}` —
  the resolved credential forwarded VERBATIM. `auth_type` is the discriminator the runtime
  dispatches on; `fields` is the opaque secret material; `config` carries non-secret
  provider config (e.g. `aws_region`, `project_id`, `region`) for env-carrier auth types.
- `setup_git_clone` in: `{repos}` — the repos to clone; the runtime reports back with
  `git_clone_complete`.

**Invariants.**
- **Decoupled credentials.** `setup_devcontainer.auth_token` is OPTIONAL. When absent the
  devcontainer command is still sent (no credential), and the runtime STILL reports ready.
  Readiness means "sandbox is operational," NOT "a credential is present."
- **Single generic credential carrier.** `setup_coding_agent_credential` is one command for
  ALL auth types. The controller does NO per-`auth_type` branching and never parses or
  interprets `fields`/`config` — the RUNTIME's dispatcher owns interpreting them and picking
  the right credential writer. Adding a new auth type is a runtime-side change; the
  controller forwards it unchanged.
- **Fixed order.** devcontainer → codex → coding-agent credential → git clone, always in
  that sequence over the one ordered connection.

**Failure modes.**
- Devcontainer `postCreateCommand` failure → the runtime reports `setup_complete` with
  `success:false` and a reason; the controller logs the failure and does NOT mark ready.
- Absent Codex bundle / coding-agent credential / git config → the corresponding command is
  simply not sent (not an error). A runtime with no coding-agent credential relies on the
  static `auth_token` path, or has no agent credential at all.

**Gotchas.**
- These commands originate at the CONTROLLER off spawn-time metadata — they are distinct
  from orchestrator-driven task relays and are not correlated request/response.
- Credential secrets (`fields`, `auth_json`) are never logged; the controller logs only the
  non-secret `auth_type` and that a send happened.
- `setup_coding_agent_credential` and the static `setup_devcontainer.auth_token` are
  parallel paths — both may be sent; the runtime decides precedence.
- Codex subscription bundle is written by the runtime only-if-absent (never clobber a bundle
  Codex may have refreshed).
