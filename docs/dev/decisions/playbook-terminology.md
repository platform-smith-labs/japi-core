# Decision: "Playbook" terminology (not "workflow") for customer-authored agentic orchestration

**Date**: 2026-07-06 (reaffirmed 2026-07-07 — resolves the ps-workflow KB-bootstrap escalation)
**Status**: Accepted
**Context**: The agentic substrate design (work-2607051522-substrate-design-of-record) needed a name
for its run entity. `workflow_run` collides three ways: (1) the legacy ps-workflow/Conductor tables
already on main (`workflow_definition`/`workflow_approval`/`workflow_notification`, migrations
0029–0036) — a `workflow_run` would read as their sibling while we plan to delete that engine;
(2) `workflow_run` is a GitHub Actions webhook event name, and the platform integrates with GitHub;
(3) Claude Code has its own Workflow tool, so "the workflow" is ambiguous inside the very sessions
running our system. `work_run` was considered and rejected — "work" is our own most-loaded word
(work items, `/work`, `work.jsonl`).

## Decision

The customer-authored orchestration bundle is a **playbook**; one execution of it is a
**playbook run** (DB entity `playbook_run`). "Workflow" is never used for our concept.

## Rules

### 1. The vocabulary

| Term | Meaning |
|---|---|
| **playbook** | The customer-authored L1 agent-config bundle (conductor command + worker skills + criteria), generated by the transformer or hand-authored |
| **`playbook_run`** | One execution: row anchoring `company_id` + A2A `conversation_id`, owning the task tree (`agent_task.playbook_run_id`), the `suspended` flag, and the bundle pin |
| **conductor** | Unchanged — the agent driving a playbook run (`/conduct` in file mode) |

```sql
-- CORRECT
CREATE TABLE playbook_run (...);
ALTER TABLE agent_task ADD COLUMN playbook_run_id ...;

-- WRONG: reads as a sibling of the legacy Conductor tables + collides with GitHub's webhook event
CREATE TABLE workflow_run (...);
```

### 2. Where "workflow" remains correct

Only for things that already own the word: the **ps-workflow** repo — **its whole domain surface, not
just its DB tables**: the entities (`workflow_definition`, `workflow_approval`, …) AND the HTTP API it
exposes (`/workflow-executions`, `workflow-approvals`, …). ps-workflow fronts Conductor OSS, whose
primitive *is* a "workflow" — a distinct domain from the product's customer-authored playbook. Also:
external systems (Temporal/GitHub Actions workflows); Claude Code's own Workflow tool; generic English
in prose ("the epic/work workflow"). Never for the customer-facing product **playbook** concept.

> **Scope ruling (maintainer, 2026-07-07).** "playbook" scopes to the **product orchestration concept**
> (the customer-authored bundle + `playbook_run`); a repo's **own domain vocabulary keeps its real
> names**. A KB/description of ps-workflow correctly uses "workflow" (the code's actual API/entity
> names) — describing it as "playbook" would be inaccurate. This resolves the ps-workflow KB-bootstrap
> escalation: the KB was right to describe the real code names, and the epistemics rule ("code over a
> directive") worked as intended. A descendant's scaffold-rules terminology mandate is the *product*
> rule; it does not override a repo's own accurate domain terms.

### 3. Reserved words (do not repurpose)

- **pipeline** — reserved for a possible future *deterministic* execution surface (research/0027);
  keeps the agentic/deterministic distinction lexically clean.
- **routine** — Claude Code's scheduled cloud agents are called routines; avoid.
- **recipe** — already used by the platform's launch/artifact machinery; avoid.

## Rationale

"Playbook" captures the essence — customer-authored, prescriptive-but-adaptive orchestration as
text — with zero collisions in our DB, in Claude Code, in GitHub, or in the framework landscape
(Ansible's playbooks are the closest association and a helpful one). The rename was made while the
entity exists only in design docs, so the cost was a documentation sweep
(research/0001–0012 of work-2607051522-substrate-design-of-record; frozen legacy copies
intentionally untouched).

## Enforcement

Code review + this doc; the design docs and the eventual migration/tool names carry the term.
