# Decision: Conduct-board `register` mode — an item may follow a status-of-record FILE instead of the phase-barrier workflow

**Date**: 2026-07-11
**Status**: Accepted
**Context**: `work-2607051522-substrate-design-of-record` is a **standing research register**: per its
seq-20 disposition it does **no** requirements/planning/implementation of its own — its whole job is
to carve points out of the design of record into child work items over time, tracked in a
hand-maintained `ticket-register.md`. But the default conduct-board workflow (and the prime-strand-row
feature, see [conduct-board-prime-strand-row.md](./conduct-board-prime-strand-row.md)) injected the
item itself as a ★ gating worker strand that perpetually "owes requirements" and dragged the barrier
back to `requirements (OPEN)` — prompting `/work` on the register, which its disposition forbids. The
board's "DO THIS NEXT" was actively misleading on every tick.

## Decision

The **default** conduct-board workflow (phase barrier + prime-strand row) is unchanged. An item may
**opt out** by declaring itself a **`register`** with a **status-of-record file**: it does no
execution, so the board suppresses its prime/self row, computes the barrier from its children only,
and points "DO THIS NEXT" at the designated file (the picked-up/finished/pending ticket list).

## Rules

### 1. Opt in via two event fields on the item's own log

```bash
# CORRECT: mark a conductor-only register + its status-of-record file
scripts/wlog.sh docs/work/<id> meta_changed board_mode=register board_file=ticket-register.md
scripts/wrender.sh docs/work/<id>

# WRONG: hand-edit manifest.md or the BOARD region to delete the ★ row
#   (generated by conduct-board.sh --write / wrender.sh — will be overwritten)
```

`board_mode`/`board_file` are read from the last `created`/`meta_changed` event carrying them
(same fold pattern as every other derived field). Absent → normal workflow.

### 2. Register mode changes the board render — nothing else

- **No prime/self row.** The register is not a gating member (`ROWS` is not prepended with
  `PARENT_REPO|PARENT_ID`), so it never "owes" a phase and never drags the barrier. The barrier is
  computed from **children only**.
- **The view = live tree + the board file's table, rendered.** Instead of the flat direct-children
  table, register mode renders the **recursive scaffolded-work-item tree** (every descendant + repo +
  live status), then renders the **first markdown table in `board_file` as a box-drawing table**
  (via `mdbox()` — display-width aware so emoji columns stay aligned) plus its `**Totals**` line.
  This is the "what was identified in research vs what's been scaffolded" view. (The board file stays
  plain editable markdown; the box is a render, not stored.)
- **Header + NEXT point at the file.** Header prints `📋 register — status of record: <board_file>`;
  "DO THIS NEXT" is `carve the next point: /conduct <id> scaffold <repo> "…"` — a register never
  "completes"; it keeps carving.

### 3. Default items are untouched

```text
# CORRECT: a normal parent (no board_mode) still shows the ★ prime row + phase workflow
runtime  work-…-a2a-delivery-tracking  validation  ★ ✅ complete (validated)
👉 🎉 Children complete — settle its own parent-facing validation …

# a register (board_mode=register) shows neither the ★ row nor the phase NEXT
📋 register — 3/3 carved ticket(s) validated; … tracked in ticket-register.md
👉 carve the next point:  /conduct <id> scaffold <repo> "…"
```

## Rationale

**Special workflows are the exception, not the rule.** Most items are execution strands and want the
barrier + prime-strand row. A research register is a genuinely different node — a conductor/index whose
truth lives in a curated file, not in phase state. Encoding that as an opt-in marker keeps the default
correct for everyone while letting a special item declare its own view, rather than hard-coding
register semantics or abusing `phase_done` no-ops (which would falsely claim the register "did"
requirements/planning it never ran).

**Derived, not hand-edited.** The board stays fully generated from the event log; the only new input is
two fields on the item's own `meta_changed` event. No manifest hand-editing, consistent with the
append-only-work-event-log law.

## Exceptions

1. **A register that later needs to execute** — flip it back by appending `meta_changed board_mode=`
   (empty) and it resumes the normal workflow.

## Enforcement

- Behavior lives in `scripts/conduct-board.sh` (`render()`), gated on `BOARD_MODE == "register"`;
  synced to every repo by `scripts/sync-workflow-tooling.sh`.
- Regression check: a normal parent still renders the ★ prime row + phase-workflow NEXT (verified on
  `work-2607090251-a2a-delivery-tracking`).
