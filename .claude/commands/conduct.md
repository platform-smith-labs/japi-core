# Conduct Command (parent/child cross-repo conductor)

**Purpose**: Coordinate multi-repo work from a **parent work item** — the successor of `/epic`.
The epic entity is removed: a work item plays the parent/conductor role, and **child work items**
(in any repo) declare `parent=<work-id>` + `parent_project=<repo>` at creation. Nesting is
**N-level** (2026-07-06): any node may parent children — program → sub-effort → per-repo strand —
with two rules: the `parent=` chain must terminate at a standalone root with **no cycles**
(validated at child-create), and a mid-level node obeys the **settling rule** (below). See
`docs/dev/decisions/parent-child-work-items-and-conduct.md`.

> **Settling rule (mid-level nodes).** A node that has children wears two hats: child of its own
> parent, conductor of its own cohort. It may **not** settle its final/validation `phase_done`
> toward its parent while its own children board is incomplete — its children's completion is its
> evidence. Earlier phases may settle on conductor judgment. The board prints the settling hint
> when a mid-level node's children complete.

> **Legacy epics are frozen, not migrated.** Anything under `docs/epics/` (and any work item with an
> `epic=` link) stays on `/epic` + `scripts/epic-board.sh`, unchanged. New cross-repo work uses a
> parent work item + `/conduct`. Never convert a legacy epic.

> ## 🧱 Rollup is DERIVED · logs are WRITER-PARTITIONED — READ FIRST
>
> - The parent manifest's board (the `**Barrier Phase**` line + children table between
>   `<!-- BEGIN BOARD -->` / `<!-- END BOARD -->`) is **generated by `scripts/conduct-board.sh
>   --write <parent-id>`** from the child logs — **never hand-edit it**. `scripts/wrender.sh`
>   preserves the region across re-renders. Membership is derived from each child's `parent=`
>   declaration — there is no hand-maintained roster.
> - **Every jsonl has exactly ONE writer** (the file-mode mirror of the product substrate's
>   ownership model):
>
>   | Log | Events | Sole writer |
>   |---|---|---|
>   | child `work.jsonl` | `created`\*, `status_changed`, `phase_done`, `artifact_added`, `relay_sent`, `relay_resolved`, `escalated`, … | the child's own tooling (`scripts/wlog.sh`) — \*`created` may be seeded by the conductor at scaffold time (the **birth exception**), then ownership hands off |
>   | child `relays.jsonl` | `relay_received`, `relay_synced` (delivery state) | **the conductor only** (`scripts/rlog.sh`) |
>   | **parent** `work.jsonl` | conductor decisions: tick notes, round-cap `escalated`, status | the conductor (it's the conductor's own item) |
>   | child `barrier.md` | the conductor-PUSHED barrier signal (phase + OPEN/HELD + the child's per-role instruction) | **the conductor only** (written on every sync) — the child READS it, never edits it |
>
>   Reads are unrestricted — `wrender.sh` and `conduct-board.sh` fold both logs.

## Command Usage

```bash
/conduct <parent-id> board            # Conductor dashboard: sync first, then barrier + ready-set + "DO THIS NEXT"
/conduct <parent-id> sync             # Deliver relays (push) + refresh derived board (conduct-board.sh --write)
/conduct <parent-id> scaffold <repo> "strand prompt"   # Create a child work item in <repo> (conductor-driven)
/conduct <parent-id> next             # Surface the next sibling parent from a roadmap artifact
/conduct list                         # List parent work items (standalone items with children)
```

`<parent-id>` may be the full `work-<YYMMDDHHMM>-<slug>` or a slug fragment — resolution is a
**glob, never arithmetic** (exact dir first, then `*fragment*`, across `docs/work/` and
`repos/*/docs/work/`; error on zero or multiple matches). There is **no epic id form** — parents are
ordinary work items minted by `/work` (see `/work` for ID rules: timestamped slug, no scan, no
counter).

> ## ⚙️ Conductor model (barrier-synchronized) — semantics preserved from /epic
>
> Children advance **one phase at a time across ALL repos**: `requirements → planning →
> implementation → validation`. **No child starts phase P+1 until every child has settled phase P
> AND all relays for phase P are resolved.** The parent's repo (normally `solution`) is the **sole
> conductor** — it owns sync and tells the human what to run where. All the hard-won `/epic` rules
> apply **verbatim**:
>
> - **Relays are immutable, append-only.** Message = an immutable file under
>   `<child>/relays/{outbound,inbound}/`. Lifecycle: `relay_sent` (child, `work.jsonl`) →
>   `relay_synced`/`relay_received` (**conductor, `relays.jsonl` via `scripts/rlog.sh`**) →
>   `relay_resolved` (target child, `work.jsonl`). Resolution is an **event, never a file move**.
> - **A1 — one resolution closes BOTH legs.** A relay is ONE thing keyed by `slug`; a single
>   `relay_resolved` anywhere among the children settles the target's inbound leg AND the source's
>   outbound leg. The source never emits its own resolution.
> - **A2 — auto-resolve informational replies at delivery.** A `confirms`/`fyi` relay needs no
>   target action — the conductor resolves it at delivery time (`wlog.sh` on the **target**:
>   `relay_resolved direction=inbound slug=…` — permitted: A2 is a conductor-performed action
>   recorded as the settled fact it is). Only `blocks` relays stay open. Never answer a
>   `confirms`/`fyi` with another one — acknowledge by resolving.
> - **Round-cap → `escalated`.** If the same ordered edge (`source→target`, by `relay_sent` events)
>   has exchanged **3+ rounds in the current phase** without settling, do NOT re-deliver — append
>   `escalated` on the **parent's** log naming the edge + open asks. Escalated = **out of play until
>   a human decides** (distinct from Blocked): the board excludes it from the barrier min, emits no
>   run-command for it, and the run cannot complete while any child is escalated. A human resumes
>   with a `status_changed` (or cancels).
> - **The barrier signal is DERIVED in the parent manifest and PUSHED to children.** The parent
>   manifest's `**Barrier Phase**` line is the source of truth — unambiguous by construction:
>   `<phase> (OPEN)` = every child may run `<phase>` now; `(HELD)` = nobody starts it. But **repo
>   isolation means a child can NEVER read the parent manifest** (maintainer, 2026-07-13) — on
>   EVERY sync the conductor writes the current barrier into each child's own tree as
>   `{child-WD}/barrier.md` (conductor-owned, like `relays.jsonl`). **Children READ their own
>   `barrier.md`; they NEVER recompute the barrier and NEVER reach across repos for it.** A child
>   whose `barrier.md` is missing or looks stale idles and asks the human to run `/conduct sync`.
> - **VALIDATION is TWO-LAYER.** Children run their **local** suites and relay
>   `to-<parent-repo>--<repo>-e2e-needs` (`kind=blocks`, `phase=validation`); the **parent's repo**
>   owns the cross-repo e2e, drives it on a live stack, resolves each needs-relay as covered+passing,
>   and settles validation only on full-e2e GREEN. `conduct-board.sh` emits the correct per-role
>   command automatically.
> - **`scripts/conduct-board.sh <parent-id>`** = read-only conductor view; **`--write`** refreshes
>   the parent manifest's board region. The script computes; only Claude (this command) does the
>   nuanced relay delivery.

## Behavior

### `/conduct <parent-id> scaffold <repo> "strand prompt"`

Create a child work item in the target repo (conductor-driven creation — the **birth exception**:
the conductor seeds the child's `created` event, then never writes that log again).

1. Resolve `<parent-id>` → `$PWD_PARENT` and validate the chain: walk the parent's own `parent=`
   links upward — the chain must terminate at a standalone root, must not revisit any node
   (**no cycles**), and must not include the child being created. Error on any violation.
2. Resolve `<repo>` via the root `CLAUDE.md` `## Repo Aliases` table.
3. Mint the child id (`work-$(date +%y%m%d%H%M)-<slug>`, slug from the strand prompt — no scan),
   `WD={repo}/docs/work/<child-id>` (or `docs/work/<child-id>` for `solution`).
4. Scaffold + seed (equivalent to `/work --parent-work` run in the child repo):
   ```bash
   mkdir -p "$WD"/{research,requirements,issues,plans,relays/outbound,relays/inbound}
   scripts/wlog.sh "$WD" created title="<title>" slug=<slug> kind=work repo=<repo> \
     owner=<owner> request="<strand prompt>" \
     parent=<parent-id> parent_project=<parent-repo> [wishlist=<n>]
   scripts/wrender.sh "$WD"
   ```
5. **Inherit scaffold rules.** Walk the child's ancestry chain root→parent; for each ancestor
   whose work dir contains a **`scaffold-rules.md`**, copy its rules into the new child's
   `context.md` (root's rules first, nearest ancestor's last) and note the source
   (`inherited from <ancestor-id>/scaffold-rules.md`). A descendant that later becomes a parent
   passes them on the same way. (This is how a program-level directive — e.g. the epistemics
   "prior research is not gospel" rule — binds every descendant automatically.)
6. Journal on the parent: `scripts/wlog.sh docs/work/<parent-id> note note="scaffolded <child-id> in <repo>"`,
   then `scripts/wrender.sh docs/work/<parent-id>`.
7. Output the resume command: `cd {repo}/ && /work <child-id>`.

The board discovers the child automatically on the next fold (its `parent=` declaration is the
membership record) — never hand-add a roster row.

### `/conduct <parent-id> board`

The conductor view — **sync first, then render**:
1. Run the full `sync` flow below (near no-op when nothing is pending).
2. Run `scripts/conduct-board.sh <parent-id>` (read-only), present its output verbatim (barrier,
   per-child states, DO THIS NEXT, the `run in each repo` block with real work ids + close
   commands), and add a one-line interpretation. **Never hand-recompute the barrier or hand-write
   per-repo commands.** For a passive monitor: `scripts/conduct-board.sh <parent-id> --watch`.

### `/conduct <parent-id> sync`

The conductor's only mutating action (children never push state up).

1. **Load**: resolve the parent; children = `conduct-board.sh`'s derived roster (or scan
   `parent=<parent-id>` declarations directly).
2. **Deliver outbound relays (push)**. For each child, glob `relays/outbound/to-*` whose slug has a
   `relay_sent` (child `work.jsonl`) but **no** `relay_synced` (child `relays.jsonl`). For each
   `to-{target}--{slug}.md`:
   - **Target repo has no child yet** → run the **scaffold** flow above for it (title/context from
     the relay), then continue.
   - Copy the message (immutable) to `{target-WD}/relays/inbound/from-{source}--{slug}.md`.
   - Record delivery — **conductor log, conductor writer**:
     ```bash
     scripts/rlog.sh "$TARGET_WD" relay_received from={source-repo} slug={slug} \
       relay_kind={kind} phase={phase} ask="<ask>" path=relays/inbound/from-{source}--{slug}.md
     scripts/rlog.sh "$SOURCE_WD" relay_synced slug={slug}
     scripts/wrender.sh "$TARGET_WD"; scripts/wrender.sh "$SOURCE_WD"
     ```
     (`relay_synced` marks delivery only — it does **not** close the relay.)
   - **A2**: if `kind` ∈ {`confirms`,`fyi`} → close now:
     `scripts/wlog.sh "$TARGET_WD" relay_resolved direction=inbound slug={slug}` + re-render.
   - **Round-cap**: 3+ rounds on this edge this phase → do NOT deliver; append `escalated` on the
     **parent** log (edge + open asks) and surface it.
3. **Refresh the derived board**: `scripts/conduct-board.sh --write <parent-id>` — recomputes the
   `**Barrier Phase**` + children table into the parent manifest's BOARD region. Do not hand-edit.
3b. **Push the barrier signal into EVERY child (mandatory, every sync).** Repo isolation means
   children cannot read the parent manifest — this push is their ONLY barrier source. For each
   child, write `{child-WD}/barrier.md` (conductor-owned; overwrite wholesale each sync):
   ```markdown
   # Barrier — conductor-pushed (DO NOT EDIT; sole writer: the conductor via /conduct sync)
   **Parent**: <parent-id> @ <parent-repo>
   **Barrier Phase**: <phase> — OPEN|HELD
   **Pushed**: <date>
   <one line: why — e.g. "all repos settled implementation, zero open relays">
   ## Your instruction (<repo> — <role>)
   <the board's per-repo run-command / two-layer validation instruction, verbatim>
   ```
   Take the phase + per-repo instruction from the fresh `conduct-board.sh` output (step 3) —
   never hand-compute them. Skip a child only if its board state is 🚨 Escalated (out of play).
4. **Journal the tick**: append one `note` on the parent log summarizing actions taken (deliveries,
   scaffolds, escalations) + `scripts/wrender.sh` the parent. This is the conductor's decision
   journal — the epic model's hand-authored Change Log, promoted to events.
5. **Reflect to the wishlist at phase boundaries** (if the parent has a `**Wishlist**` field):
   refresh the wishlist item's Tracking row from the board output — derived, never per-event
   hand-sync (same contract as `/epic`; see the `wishlist` skill).
6. Present: deliveries made, the board output, next actions.

### `/conduct <parent-id> next`  — sequenced sibling parents (roadmaps)

3-level nesting (epic-of-epics) is retired. A multi-milestone effort is a **chain of sibling
standalone parents**, sequenced by a roadmap:
- **Wishlist-derived**: the wishlist item's `## Tracking` table / roadmap is the sequence (same as
  `/epic next-milestone`) — confirm the current parent is ✅ Completed, pick the first
  unscaffolded milestone, create the next **standalone parent** via `/work "<milestone title>"`
  (carrying `wishlist=<n>`), and update the wishlist Tracking row + registry (bidirectional link,
  same contract as `/epic`'s Wishlist Linkage).
- **Standalone roadmap**: a "vision" parent may carry a `## Roadmap` artifact (a plans/ or
  research/ doc) listing the sequence; `next` reads it, confirms the current row's parent is
  Completed, and scaffolds the next sibling parent, back-linking both ways in prose.

### `/conduct list`

Glob all `docs/work/*/work.jsonl` + `repos/*/docs/work/*/work.jsonl`; a **parent** is any standalone
item that at least one child's log names in a `parent=` declaration. Show id · repo · title ·
barrier phase (from its manifest board, if rendered) · # children.

## Monorepo Root Resolution

Same as `/epic`: run from the monorepo root, or resolve it — `./docs/work/` + `./repos/` present ⇒
CWD is the root; else if `../repos/` exists ⇒ root is `..`. Repo names always resolve via the root
`CLAUDE.md` `## Repo Aliases` table.

## Integration with Other Commands

- `/work "prompt"` — create a **standalone** item (a potential parent — parenthood is implicit; it
  becomes a parent when its first child is created).
- `/work --parent-work <parent-id> --parent-project <repo> "prompt"` — create a **child** directly
  from within its repo (both flags present, or both absent). Same effect as `scaffold`. The parent
  may itself be a child (N-level); the create-time chain validation applies.
- `/work <child-id>` / `/work <child-id> auto` — resume/execute a child strand. Parent-bound
  children follow the conductor rules in `/work` ("Conductor-aware work"): read the parent
  manifest's `**Barrier Phase**`, never recompute it, settle with `phase_done`, STOP.
- `/epic …` — **legacy only** (frozen epics under `docs/epics/`). Do not use for new work.
- `wishlist` skill — items are picked up by a parent work item now; `/conduct` maintains the
  back-link (Tracking row + registry), one **sibling parent per milestone**.
