# KB bundle layout

The `docs/kb/` tree a repo's KB occupies. Generated by `kb-bootstrap`; `index.md` files are
projections (never hand-edited). See [schema.md](./schema.md), [kb-config.md](./kb-config.md).

## Tree

```
docs/kb/
  index.md            # GENERATED projection (kb-render.sh) — never hand-edited; carries a deterministic "Last synced: <commit-date> @ <sha>" freshness stamp
  log.md              # change history, newest-first (one entry per generation run)
  kb-config.yaml      # repo-owned STEERING file — hand-edited, survives regeneration
  self/               # this repo's canonical self-brief — REGENERATED by kb-bootstrap
    overview.md       # single file: what this repo is, its role for peers
    context.md        # single file: system context — who it talks to, ubiquitous data facts
    capabilities/     # ★ collection: ONE FILE PER FUNCTIONALITY (the point of the KB)
    interfaces/       # collection: one exposed/consumed interface per file (contract refs + behavior)
    gotchas/          # collection: cross-cutting integrator traps not tied to one capability
    decisions/        # collection: one-line ADR summaries (consequence for a peer), referencing docs/dev/decisions/
    glossary.md       # single file: domain terms a peer needs
    extract/          # GENERATED deterministic grounding fact sheet (kb-extract.sh) — always rebuilt
    eval/             # GENERATED per-run acceptance records (task-scoping eval) — excluded from concept/lint/render sweep, like extract/
  peers/<repo>/       # RESERVED for the kb-sync sibling — bootstrap creates the dir + README stub only
  notes/              # owning repo's own annotations — NEVER touched by regeneration
```

## Single file vs collection (hybrid shape)

- **Singular narrative concerns = single file**: `overview.md`, `context.md`, `glossary.md`.
- **Enumerable concerns = collection directory** of one-concept-per-file + a generated per-directory
  `index.md`: `capabilities/`, `interfaces/`, `gotchas/`, `decisions/`.

### Growth rule (avoids scaffolding noise in thin repos)

A collection MAY start as a single file (e.g. `gotchas.md`) and MUST be promoted to a directory once
it exceeds ~3 items or ~300 lines. `kb-lint.sh` accepts both forms. **`capabilities/` is always a
directory** — it is the point of the KB.

## Writer partition (who writes what)

| Path | Writer |
|---|---|
| `self/**`, `index.md`, `log.md` | `kb-bootstrap` (regeneration) |
| `self/extract/**` | `kb-extract.sh` (wholesale every run) |
| `self/eval/**` | `kb-bootstrap` (per-run acceptance record; one file per run, keyed by HEAD sha) |
| `peers/**` | reserved — the future `kb-sync` sibling |
| `notes/**` | humans / the repo's own agent (annotations) |
| `kb-config.yaml` | maintainer (hand-edited) |

## Regeneration layering (idempotence)

- `self/extract/**` — regenerated **wholesale** every run (deterministic → no churn).
- `self/eval/**` — one record **appended per run** (filename keyed by HEAD short-sha); never
  drift-managed, never linted/rendered as a concept. A fresh run at a new HEAD adds a new record; it
  does not rewrite prior records, so eval history for this repo accumulates like `log.md`.
- Narrative concepts (`capabilities/`, `interfaces/`, `overview`, `context`, `gotchas/`) — regenerated
  **only where their `evidence` drifted** (a cited path changed since `commit_sha`), keeping diffs
  small and reviewable.
- `peers/`, `notes/`, `kb-config.yaml` — **never** touched by regeneration.

## Seams for the sibling work items (do not build here)

- `self/` in aggregate = the **relay-able self-brief unit** (the future `kb-sync` relays it to peers).
- `peers/<repo>/` = `kb-sync`'s **fold target** for received briefs.
- `index.md` = the **consolidation** skillset's read surface.
